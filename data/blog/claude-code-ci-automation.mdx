Most engineers discover Claude Code as an interactive tool. You open a terminal, describe a task, and work back and forth. That is the obvious path.

The less obvious path is running Claude Code non-interactively — in CI pipelines, GitHub Actions, scheduled jobs, or as part of a larger automation chain. Anthropic documents this headless mode explicitly, and it opens up a different category of use case: automated PR review, docs generation, issue triage, test failure analysis, and more.

This article is about that category. The angle is practical and safety-focused: what actually works in CI, how to structure skills for automation, and where to keep humans in the loop.

## 1. What headless mode means

In interactive mode, Claude Code runs in a REPL-style loop: you type, Claude responds, you respond back. That loop is what most tutorials show.

Headless mode breaks that loop. Claude Code runs a single task, produces output, and exits. No back-and-forth, no prompts waiting for keyboard input. The result goes to stdout or a file, and the CI step succeeds or fails like any other shell command.

Anthropic documents this as the `--print` flag (also written `-p`), which prints the response and exits immediately. That one flag is what makes Claude Code usable in pipelines.

This distinction matters because it changes how you think about the task design:

- Interactive tasks can be exploratory and iterative
- Headless tasks must be well-scoped, self-contained, and produce deterministic enough output to be useful downstream

---

## 2. The core flags for non-interactive use

The key flags for headless use:

**`--print` / `-p`**

Runs Claude with the given prompt, prints the response to stdout, and exits. This is the foundation of all pipeline use.

```bash
claude --print "Summarize the changes in the last commit"
```

**`--output-format`**

Controls how output is structured. Options include `text` (default) and `json`. For pipelines where downstream steps need to parse output, `json` is usually preferable.

```bash
claude --print --output-format json "Review the diff for security issues"
```

**`--allowedTools`**

Restricts which tools Claude can use during the run. Critical for CI safety — you do not want a docs-generation step to accidentally make commits.

```bash
claude --print --allowedTools "Read,Grep,Glob" "Find all API handlers and list their routes"
```

**`--no-permission-prompts`**

Suppresses interactive permission prompts. In a CI context where there is no terminal to respond to prompts, this is necessary. Used together with `--allowedTools` to define a safe permission boundary upfront.

**`--max-turns`**

Caps how many agentic turns Claude takes. Useful for keeping CI steps bounded and predictable.

## 3. Authentication in CI

Claude Code uses the `ANTHROPIC_API_KEY` environment variable for authentication in headless mode. There is no interactive login flow in CI.

The pattern in GitHub Actions:

```yaml
- name: Run Claude Code analysis
  env:
    ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
  run: |
    claude --print --allowedTools "Read,Grep,Glob" \
      "Review the changed files for security issues and output findings as a bullet list"
```

Store the key as a repository secret. Never hardcode it. Never echo it in logs.

For team repositories, use organization-level secrets to avoid per-repo key management. If you use Claude on the Teams or Enterprise plan, follow Anthropic's guidance on API key scoping for your plan tier.

## 4. Safe patterns: read-only vs write-with-review

This is the most important design decision in Claude Code automation: **what should Claude be allowed to do automatically, and what should require human review?**

The clearest framework is a two-tier model:

**Tier 1: read-only analysis (safe to automate fully)**

These run on every PR or push and post results as comments or check annotations. Claude reads, analyzes, and reports. It never writes.

Examples:
- security review of changed files
- test coverage gap detection
- unused import or dead code flags
- API documentation drift detection (does the code match the docs?)
- type inconsistency scanning

Tools needed: `Read`, `Grep`, `Glob` — nothing else.

**Tier 2: write-with-review (automate the draft, not the merge)**

Claude generates a proposed change — a docs update, a changelog entry, a migration plan, a test stub — and opens a PR or posts a draft comment. A human reviews before anything merges.

Examples:
- auto-generated CHANGELOG entries from commits
- PR description drafts posted as PR body suggestions
- docs updates proposed as a draft PR
- test stubs generated from a failing test report

Tools needed: `Read`, `Grep`, `Glob`, `Write`, sometimes `Bash(git *)`.

The key discipline: **write-with-review means nothing merges without a human sign-off**. The automation produces the artifact; the human decides whether to use it.

## 5. Where human approval still belongs

Automation pressure is real. The temptation is to keep expanding what Claude does automatically. Resist it in these areas:

**Deployments**

Never automate deploys through Claude Code without an explicit human gate. Even if Claude generates the deploy command correctly 95% of the time, the 5% failure case in production is not acceptable. Gate all deploys behind a human approval step.

This aligns with Anthropic's guidance on `disable-model-invocation: true` for skills with side effects — the same principle applies in CI.

**Merges to main**

Claude can propose, draft, and suggest. It should not merge to main autonomously. Require at minimum one human review on all PRs that Claude Code touched.

**Database migrations**

Automated migration generation is useful. Automated migration execution in production is not. Keep execution a manual step.

**Anything involving credentials, secrets, or permissions**

Any task that reads, generates, rotates, or modifies credentials should require explicit human action. Do not route these through automated Claude Code steps.

**Irreversible operations**

If an action cannot be undone in under five minutes, it should not be automated without human approval. Deletions, bulk updates, external API calls with side effects — these all need a gate.

The pattern to enforce this in GitHub Actions is a required approval step or environment protection rule. Use those features — they exist exactly for this purpose.

## 6. GitHub Actions: a basic integration pattern

Here is a practical starting point for a PR review workflow:

```yaml
name: Claude Code PR Review

on:
  pull_request:
    types: [opened, synchronize]

jobs:
  claude-review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Get changed files
        id: changed
        run: |
          echo "files=$(git diff --name-only origin/${{ github.base_ref }}...HEAD | tr '\n' ' ')" >> $GITHUB_OUTPUT

      - name: Run Claude review
        id: review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude --print \
            --allowedTools "Read,Grep,Glob" \
            --output-format json \
            "Review these changed files for security issues, correctness bugs, and notable risks: ${{ steps.changed.outputs.files }}. Output a brief summary with severity labels." \
          > review_output.json

      - name: Post review comment
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const output = JSON.parse(fs.readFileSync('review_output.json', 'utf8'));
            await github.rest.issues.createComment({
              owner: context.repo.owner,
              repo: context.repo.repo,
              issue_number: context.issue.number,
              body: `## Claude Code Review\n\n${output.result}`
            });
```

This is read-only: Claude analyzes and posts a comment. It never writes to the repository.

## 7. Skills designed for automation

Skills work in headless mode. You can invoke them with `/skill-name` in the prompt passed to `--print`, or write the prompt to match the skill's description for automatic loading.

But skills designed for interactive use need adjustment for CI. The key differences:

**1. No interactive clarification**

In CI, Claude cannot ask a follow-up question. The skill prompt must be self-contained. Design skills so that all required context is either passed as `$ARGUMENTS` or derivable from the codebase without user input.

**2. Structured output**

Interactive skills can be conversational. CI skills should produce structured, parseable output. Use explicit output formatting instructions in the skill body.

Example — interactive skill body:

```markdown
Review the code and let me know what you find. Feel free to ask if you need more context.
```

Example — CI-adapted skill body:

```markdown
Review the code changes in $ARGUMENTS.

Output format (strict):
## Security
- [SEVERITY] Finding description (file:line)

## Correctness
- [SEVERITY] Finding description (file:line)

## Performance
- [SEVERITY] Finding description (file:line)

If no issues found in a category, write "None found."
Do not ask clarifying questions. Work with what is available.
```

**3. Bounded scope**

CI steps should complete quickly and predictably. Keep skill instructions focused on a single, well-defined task. Avoid instructions that could cause unbounded exploration.

**4. `allowed-tools` aligned with CI permissions**

A skill that needs `Bash` for git operations locally may need those permissions restricted in CI. Be explicit about what the CI version of the skill can touch.

## 8. Structuring output for downstream steps

When Claude Code is one step in a multi-step pipeline, output structure matters. A prose paragraph is hard to parse; a structured format is not.

**Option 1: JSON output**

Use `--output-format json` and instruct Claude to produce structured data:

```bash
claude --print --output-format json \
  "Analyze the test failures in the attached log and return a JSON object with keys: affected_files (array), likely_cause (string), suggested_fix (string)"
```

**Option 2: Markdown with consistent headers**

For GitHub comments, markdown with predictable section headers is easy to work with downstream:

```
## Critical
...

## Warnings
...

## Info
...
```

**Option 3: Machine-readable inline markers**

For steps where you only need a pass/fail signal:

```
RESULT: PASS
RESULT: FAIL — [reason]
```

Your CI step can then grep for `RESULT:` and act accordingly.

The general rule: design the output format in the skill or prompt before you design the downstream parsing. Start from what the next step needs and work backwards.

## 9. Controlling tool permissions in automated runs

In CI, tool permissions are a safety boundary, not just a feature. Use them deliberately.

**Read-only analysis steps**

```bash
claude --print --allowedTools "Read,Grep,Glob" ...
```

No writes, no shell commands. This is the safest mode for analysis tasks.

**Write-to-working-directory steps**

```bash
claude --print --allowedTools "Read,Grep,Glob,Write" ...
```

Claude can write files but cannot run shell commands or make git operations.

**Draft PR steps**

```bash
claude --print --allowedTools "Read,Grep,Glob,Write,Bash(git add *),Bash(git commit *),Bash(git push *)" ...
```

Scoped bash permissions mean Claude can make a commit but cannot run arbitrary shell commands.

**Never in CI:**

```bash
# Do not do this in CI
claude --print --allowedTools "Bash" ...
```

Unrestricted bash access in a CI context is a security risk. Scope it.

Anthropic's permission system allows `Bash(pattern)` syntax for scoping specific commands. Use it. The pattern `Bash(git *)` allows git commands but not `rm -rf` or `curl`.

## 10. Practical automation use cases

These are the use cases that tend to work well in practice:

**Automated PR review comments**

Analysis-only. Claude reviews changed files and posts findings. No writes to the repo.

Best for: security scanning, correctness checking, docs drift detection.

**Changelog generation**

Claude reads recent commits and generates a CHANGELOG entry. Posts it as a draft PR or PR comment for human review before merge.

Best for: release preparation, keeping changelogs maintained.

**Test failure triage**

Claude reads a test failure log, traces the relevant code path, and posts a structured diagnosis. The engineer reads the diagnosis and decides what to fix.

Best for: flaky test investigation, CI failure triage.

**API documentation drift**

Claude compares the current codebase to the existing API docs and flags mismatches. Useful for keeping docs in sync without manual audits.

Best for: public API maintenance, OpenAPI spec validation.

**Dependency review**

Claude reads newly added dependencies and summarizes what they do, their license, and any obvious security flags. Posted as a PR comment.

Best for: supply chain hygiene, code review assistance.

**Issue triage**

Claude reads a new GitHub issue, tries to find the relevant code, and posts a comment with context. Helps maintainers respond faster.

Best for: open source projects, large repositories with high issue volume.

## 11. Anti-patterns to avoid

**Giving Claude write access without a review gate**

If Claude writes to the repo and the change auto-merges, you have removed human review from the loop. Always require a PR + human approval for any automated write.

**Running without scoped tool permissions**

Unrestricted tools in a CI context mean that a prompt injection attack in any file Claude reads could escalate. Scope permissions to the minimum the step actually needs.

**Prompt injection via user-controlled input**

If your CI step includes PR title, PR body, issue text, or commit message content directly in the Claude prompt, a malicious actor can inject instructions. Sanitize or isolate user-controlled content. Never interpolate untrusted text directly into the system prompt.

This is the most underappreciated risk in Claude Code CI integration.

**Automating before standardizing**

If your team does not have consistent code structure, naming conventions, or test patterns, automated Claude Code analysis will produce inconsistent and noisy output. Standardize first. The [team productivity guide](/blog/claude-code-team-productivity) covers this ordering in detail.

**Treating all CI steps as equivalent**

Read-only analysis and write operations have different risk profiles. Treat them differently in terms of review gates, permission scoping, and failure handling.

**Ignoring cost**

Each Claude Code CI run costs tokens. On a busy repository with dozens of PRs per day, costs add up. Be deliberate about which events trigger which Claude Code steps. Not every commit needs a full security review.

## 12. The right mental model for CI use

Think of Claude Code in CI as an automated analyst on the team.

It reads code, produces structured findings, and drafts artifacts. It does not make decisions. It does not merge things. It does not deploy anything. It informs the humans who do those things.

That constraint is not a limitation — it is the correct scope.

The patterns that work best in CI are:

- **Read → report**: Claude analyzes, posts a comment, done.
- **Read → draft → review → merge**: Claude proposes a change, a human approves it.
- **Read → triage → notify**: Claude provides context, a human takes action.

The patterns that cause problems are:

- **Read → write → merge**: removing the human from the loop.
- **Read → deploy**: automating irreversible operations.
- **Unrestricted access**: not scoping tool permissions.

Keep Claude in the analyst seat. Keep humans in the decision seat.

That split makes CI automation with Claude Code reliable, safe, and genuinely useful without creating the kind of autonomous-system risks that make engineering teams nervous.
